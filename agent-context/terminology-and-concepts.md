# Terminology and Concepts — Zühlke Managed Services

## Business Model Terms

| Term | Definition |
|---|---|
| **MS / Managed Services** | Zühlke's business unit and service offering for ongoing, outcome-based management of client digital assets. Distinct from project delivery. |
| **AMS** | Application Management Services. Ongoing support and maintenance for live applications. Takes over as development peaks. |
| **CMS** | Cloud Managed Services. Cloud infrastructure operations, starting early in the product lifecycle and running continuously. |
| **T&M** | Time and Materials. The billing model used in project delivery. MS uses a subscription model instead. |
| **Subscription model** | MS pricing model — recurring monthly or quarterly fee in exchange for guaranteed service outcomes and defined SLAs. |
| **Innovate, Build, and Operate** | Zühlke Group's corporate strategy for 2025–2027. MS is the "Operate" pillar. |
| **Protect → Evolve → Transform** | The three-level MS service model: keep things running, improve over time, advance with AI-enhanced operations. |
| **GDC** | Global Delivery Center. Zühlke's offshore/nearshore engineering resource pool. Used for cost-effective staffing of MS engagements. Language restrictions (e.g., German required) may limit GDC use. |

## Roles

| Term | Definition |
|---|---|
| **SM / Service Manager** | Operational lead for a specific MS engagement. Primary client contact. Manages SLA compliance, incident coordination, reporting, and change governance. |
| **MS Commercial Lead** | End-to-end owner of deal commercials, contracts, SLAs, and bid governance. Single-threaded accountable owner across all deal threads. |
| **MS Operations Lead** | Leads day-to-day MS operations across all engagements. Currently Marko Simic. |
| **Account Lead** | Owns the client relationship and deal entry. Primary escalation point. |
| **Delivery Lead** | Manages project handover to MS. Involved in Operations Readiness sign-off. |
| **EM** | Engagement Manager. Involved in risk and ISO 27001 classification decisions. |
| **SDL** | Service Delivery Lead. Also referred to as the lead developer/architect in the project team. Consulted during pre-sales on technical complexity. |
| **SPOC** | Single Point of Contact. One dedicated SM is the SPOC per engagement. |

## SLA & Service Concepts

| Term | Definition |
|---|---|
| **SLA** | Service Level Agreement. The contractual commitment for response and resolution times by severity. |
| **SLT** | Service Level Target. Specific measurable targets within an SLA. |
| **BC / Business Criticality** | Assessment of how important a system is to a client's operations. Drives SLA tier. Levels: Low, Medium, High, Very High. |
| **Response Time** | Measured from ticket open to acknowledgement. |
| **Resolution Time** | Measured from ticket open to ticket closed. |
| **Severity 1** | Critical — system down, no workaround. 24×7. 15 min response / 4 hr resolution. |
| **Severity 2** | High — major impairment, workaround inadequate. Business hours. 2 hr response / 2 BD resolution. |
| **Severity 3** | Medium — partial impairment, acceptable workaround. Business hours. 8 BH response / 5 BD resolution. |
| **Severity 4** | Low — minor issue. Business hours. 1 BD response. |
| **Standard tier** | 09:00–17:00, working days. |
| **Extended tier** | 06:00–20:00, working days. |
| **24×7 tier** | 00:00–24:00 every day including public holidays. |
| **Hypercare** | Elevated support post-go-live for a defined period — faster response, dedicated team attention, closer monitoring. A service model choice, not a legal obligation. |
| **Warranty (Gewährleistung)** | Legal obligation about liability. Typically 2–4% of initial project investment, budgeted into the base fee. Not the same as Hypercare. |
| **Bonus/malus** | Contract mechanism where Zühlke has both downside (penalty) and upside (bonus) tied to service performance. Zühlke's preferred alternative to penalty-only clauses. |

## Process Terms

| Term | Definition |
|---|---|
| **Gate 0/1** | Mandatory bid governance checkpoints. Gate 0 = approved contract template selected. Gate 1 = SLAs defined + cost plan reviewed. No contracting without all gates cleared. |
| **TZW** | Acceptance process agreed with client during onboarding. Defines who fixes findings from the project handover phase. |
| **Operations Readiness Checklist** | Internal sign-off that handover from project to MS is complete. Focus on Documentation, Code, Infrastructure. |
| **Client Readiness Checklist** | Confirms all client contributions are in place before the MS engagement starts. Replaces verbal confirmations. |
| **QBR** | Quarterly Business Review. Structured client review covering SLA performance, trends, risks, and improvement plans. |
| **CSI** | Continual Service Improvement. Formal process for identifying and implementing improvements. Tracked and reported. |
| **RACI** | Responsible, Accountable, Consulted, Informed. Used to define role ownership across MS processes. |
| **SMS** | Service Management System. The formal documentation set for all MS processes, stored on SharePoint. Required by ISO 20000. |
| **ITSM** | IT Service Management. The discipline and tooling (JSM, ServiceNow, FreshService) used to manage incidents, changes, and service requests. |

## Commercial Terms

| Term | Definition |
|---|---|
| **DACH** | Germany (D), Austria (A), Switzerland (CH). Primary MS market. German-language contracts and client comms are standard. |
| **Base MS** | The core MS contract template. Used for every deal. |
| **Market Addendum** | Country-specific legal annex attached to the Base MS template (CH, DE, AT, UK, or SG). |
| **Language Plan** | Document defining which roles must be German-speaking and where English is acceptable. Used for DACH clients with mixed-language teams. |
| **BidCalc** | A bidding calculation tool (referenced in Glossary). Used to calculate pricing. |
| **Service Calculator** | Private app for calculating service fees based on scope, SLA tier, and support hours. Available on demand. |
| **Vertec** | Time tracking and project management tool. Used for budget and resource reporting. |

## Technical Terms

| Term | Definition |
|---|---|
| **PUROZ** | Referenced in Glossary. Internal term (exact meaning: see Glossary page). |
| **TERRA Model** | Referenced in Glossary. Internal model used in MS context. |
| **KEDB** | Known Error Database. Used in incident and problem management. |
| **First Time Fix Rate** | KPI measuring the percentage of incidents resolved on first contact/attempt. |
| **KPI** | Key Performance Indicator. All measured KPIs add cost — during pre-sales, verify the client is willing to pay for each one tracked. |
| **Release Management** | Process for managing and deploying application releases within the MS engagement. |
| **Change Request** | Formal request to modify the system or service. Managed through the change process. |
| **Incident** | Unplanned interruption or reduction in quality of a service. Triggers SLA response clock. |
| **Problem** | Underlying cause of one or more incidents. Managed separately from incidents. |
| **Runbook** | Documented procedures for handling specific operational scenarios. Created during onboarding and maintained by the support team. |
| **ISO 20000** | International standard for IT Service Management. Zühlke MS is certified. |
| **ISO 27001** | International standard for Information Security Management. All MS projects must comply. |
| **System Complexity Assessment** | Framework for scoring technical complexity of the application being managed. Drives effort and pricing estimates. |
